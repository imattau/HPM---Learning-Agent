# Sub-project 7: PhyRE ‚Äî Physics Reasoning Environment

## Goal

Extend the HPM benchmark suite with a physics reasoning environment (PhyRE) that tests whether the hierarchical abstraction stack generalises from symbolic domains (ARC, math) to continuous 2D physics. PhyRE uses pre-computed pymunk simulations to avoid runtime physics costs. Agents must place an action object (ramp, bumper, surface, or ball) so that an active ball reaches a circular goal region after 2 seconds. L4 and L5 from SP6 are reused without modification ‚Äî PhyRE validates their domain-agnostic design.

## Architecture

### Physics Simulation (`benchmarks/phyre_sim.py`)

All simulation runs at **generation time only**. Scoring at benchmark time reads pre-computed snapshots from `data/phyre_tasks.pkl`.

**Scene layout:**
- 2D box: 500√ó500 units, static walls on all four sides
- 1‚Äì3 dynamic balls per scene
- Goal: circular target region (radius=30 units), position fixed per task
- Active ball: the ball whose entry into the goal region is the success criterion

**Simulation parameters:**
- Fixed timestep: `dt = 1/60 s`
- Duration: 120 steps = 2 seconds of simulated time
- Gravity: `(0, -500)` units/s¬≤
- Seed: 42 for all random scene generation (fully deterministic)

**Physics families and action objects:**

| Family | Action object | Dominant mechanism |
|---|---|---|
| Projectile | Ramp (static segment) | Deflects ball; gravity dominates trajectory |
| Bounce | Bumper (circular, high restitution) | Elastic wall; restitution dominates |
| Slide | Surface/ramp (high friction) | Friction dominates; low-speed lateral motion |
| Collision | Ball (dynamic, medium mass) | Momentum transfer; collision dynamics dominate |

**pymunk API used:** `pymunk.Space`, `pymunk.Body` (STATIC / DYNAMIC), `pymunk.Circle`, `pymunk.Segment`, `pymunk.Shape.elasticity`, `pymunk.Shape.friction`, `pymunk.moment_for_circle`, `Space.step(dt)`

**Goal checker:** After each `Space.step`, test whether active ball centre is within 30 units of goal centre. Task succeeds if this holds for at least 1 step during the simulation.

**Task generation:** For each (family, scene_index) pair, generate 5 candidate placements:
- 1 correct placement (active ball reaches goal)
- 4 distractors: wrong-family action objects applied to the **same** `test_initial` snapshot. This is the critical constraint ‚Äî all 5 candidates share an identical initial scene state, so the only variable is the action object type and position.

### Task Schema

Tasks are stored as a list of dicts serialised to `data/phyre_tasks.pkl` (trusted internal data; generated by our own code from seed=42).

```python
{
    "task_id": "bounce_042",          # "{family}_{index:03d}"
    "family": "Bounce",               # one of Projectile|Bounce|Slide|Collision
    "train": [                        # 3‚Äì4 (init, final) snapshot pairs
        {"init": [...], "final": [...]},
        ...
    ],
    "test": {
        "init": [...],                # test_initial snapshot (shared by all candidates)
        "candidates": [               # exactly 5 candidates
            {
                "action": {           # action object parameters
                    "family": "Bounce",
                    "x": 240.0,
                    "y": 310.0,
                    "radius": 15.0,
                    "restitution": 0.95,
                    "friction": 0.1,
                    "mass": 2.0
                },
                "final": [...]        # final snapshot after simulation
            },
            ...
        ],
        "correct_idx": 2              # index into candidates list
    }
}
```

**Snapshot format:** A snapshot is a list of object dicts, one per object in the scene:

```python
{
    "id": 0,                  # object index (0 = active ball)
    "x": 120.5, "y": 340.2,  # position (units)
    "vx": 0.0, "vy": 0.0,    # velocity (units/s)
    "mass": 1.0,              # kg
    "radius": 12.0,           # units (for balls)
    "restitution": 0.8,
    "friction": 0.3,
    "is_goal": False,         # True for goal region marker
    "is_action": False        # True for action object
}
```

**Dataset scale:** 4 families √ó 60 tasks = 240 tasks total.

### Encoders (`hpm/encoders/phyre_encoders.py`)

All encoders implement the `LevelEncoder` protocol: `feature_dim: int`, `max_steps_per_obs: int = 1`, `encode(obs) -> np.ndarray`.

The `obs` argument to `encode` is a `SceneSnapshot` dataclass defined in the same file:

```python
@dataclass
class SceneSnapshot:
    objects: list[dict]    # list of object dicts (as defined above)
    goal_x: float
    goal_y: float
    goal_radius: float = 30.0
```

---

**`PhyreL1Encoder`** ‚Äî Kinematic deltas (sensory regularities)

`feature_dim = 16`, `max_steps_per_obs = 1`

Encodes the observable motion of objects between `init` and `final` snapshots. Input is a pair `(init_snapshot, final_snapshot)` as `SceneSnapshot`.

Feature construction (per non-goal object, padded to 3 objects):
- `Œîx / 500`, `Œîy / 500` ‚Äî normalised position change
- `Œîvx / 200`, `Œîvy / 200` ‚Äî normalised velocity change (max_v = 200)
- Pad with zeros if fewer than 3 dynamic objects present

Padding: 3 objects √ó 4 features = 12 features

Additional features:
- Active ball distance to goal at `init`: `dist_init / 500`
- Active ball distance to goal at `final`: `dist_final / 500`
- Action object position: `action_x / 500`, `action_y / 500`

Total: **16 features**

Distance: Euclidean from ball centre to goal centre.
Active ball: object with `id == 0`.
Action object: object with `is_action == True`.

---

**`PhyreL2Encoder`** ‚Äî Material properties (latent structural representations)

`feature_dim = 14`, `max_steps_per_obs = 1`

Encodes the physical properties of objects (mass, restitution, friction) ‚Äî invariant across time. Input is `init_snapshot` only (properties do not change during simulation).

Feature construction (per non-goal object, padded to 3 objects):
- `mass / 10`, `restitution`, `friction`
- Pad with zeros if fewer than 3 dynamic objects

Padding: 3 objects √ó 3 features = 9 features

Action object properties:
- `action_mass / 10`, `action_restitution`, `action_friction` = 3 features

Epistemic signal from L1 (passed in at encode time):
- `l1_weight: float` ‚Äî L1 pattern weight (from pattern store)
- `l1_loss: float` ‚Äî L1 NLL score for this pair

Total: **14 features**

---

**`PhyreL3Encoder`** ‚Äî Physical law summaries (relational rules)

`feature_dim = 12`, `max_steps_per_obs = 1`

Encodes summary statistics about the physics of the simulated trajectory. Input is a pair `(init_snapshot, final_snapshot)`.

Features:
1. Total momentum change magnitude (normalised by `mass_sum √ó 200`)
2. Energy dissipation ratio: `1 ‚àí KE_final / max(KE_initial, 1e-8)`, clamped to `[0, 1]`
3. Net displacement of active ball ‚Äî magnitude: `displacement / 500`
4. Net displacement of active ball ‚Äî direction: `angle / œÄ` (atan2(Œîy, Œîx), range [‚àí1, 1])
5. Goal achieved flag: `1.0` if active ball entered goal region, else `0.0`
6. Max velocity attained by active ball (normalised by `200`)
7. Collision count (total across all dynamic objects, normalised by `10`)
8. Action object mass/restitution ratio: `(action_mass / 10) √ó action_restitution`
9. Epistemic from L2: `l2_weight`
10. Epistemic from L2: `l2_loss`

Total: **10 explicit physics features + 2 epistemic = 12 features**

**Energy dissipation formula:**
```
KE_t = 0.5 √ó Œ£_i (mass_i √ó (vx_i¬≤ + vy_i¬≤))
dissipation = 1 ‚àí KE_final / max(KE_initial, 1e-8)
dissipation = clamp(dissipation, 0.0, 1.0)
```

Collision count: tracked during simulation by attaching a collision handler to `pymunk.Space` at generation time. Stored in final snapshot metadata field `"n_collisions"`.

### Benchmark Conditions (`benchmarks/structured_phyre.py`)

Four conditions evaluated on the same 240 tasks:

| Condition | Encoding | Œ≥ (scoring interpolation) | Description |
|---|---|---|---|
| `flat` | L1 snapshot only | ‚Äî | Baseline: 2-agent partitioned training, no hierarchy |
| `l2l3` | L2 + L3 scoring | Œ≥ = 1.0 fixed | Hierarchical baseline (consistent with math/ARC) |
| `l4_only` | L2 + L3 + L4 | Œ≥ = 1.0 always | L4 intuition without L5 supervision (ablation) |
| `l4l5_full` | L2 + L3 + L4 + L5 | Œ≥ from L5 | Full stack with adaptive strategy selection |

**Condition: `flat`**
- Two agents: Agent-A trained on families {Projectile, Bounce}, Agent-B on {Slide, Collision}
- Each agent uses L1 snapshot features only
- Scoring: cosine similarity between test candidate L1 feature and mean training L1 feature
- No hierarchy, no L2/L3 signals

**Conditions `l2l3`, `l4_only`, `l4l5_full`**
- Use `StructuredOrchestrator` with `PhyreL1Encoder`, `PhyreL2Encoder`, `PhyreL3Encoder`
- `l4_only`: instantiate `L4GenerativeHead(feature_dim_in=14, feature_dim_out=12)`, skip `L5MetaMonitor`
- `l4l5_full`: instantiate both `L4GenerativeHead` and `L5MetaMonitor` (no parameter changes from SP6)
- Scoring: as defined in SP6 (`Œ≥ √ó l4_score + (1‚àíŒ≥) √ó l3_nll`)

**L4/L5 reuse:** `L4GenerativeHead` and `L5MetaMonitor` from SP6 are imported without modification. PhyRE's L2/L3 feature dimensions differ from math (14 and 12 vs 10 and 12), but both classes accept arbitrary `feature_dim_in` / `feature_dim_out` ‚Äî no code changes needed.

**Accuracy metric:** fraction of tasks where `argmin(score)` equals `correct_idx`.

**Expected results:**
- `flat`: lowest accuracy (no physical law abstraction)
- `l2l3`: material property + law-level discrimination improves over flat
- `l4_only`: tests whether material properties alone predict physical law summaries
- `l4l5_full`: should match or exceed `l2l3`; L5 detects when L4 misfires across physics families

## Testing

**Encoder unit tests (`tests/encoders/test_phyre_encoders.py`):**
- L1: given two snapshots with 2 objects, output shape is `(16,)`, all values in `[‚àí1, 1]` after normalisation
- L1: with 1 dynamic object, unused slots are zero-padded
- L2: output shape is `(14,)`, restitution and friction values pass through unchanged
- L3: dissipation formula correct ‚Äî zero initial KE yields `dissipation=0.0` (clamped), not NaN
- L3: goal achieved flag is `1.0` when active ball displacement puts it within goal radius
- All encoders: `feature_dim` attribute matches actual output dimension

**Benchmark smoke tests (`tests/benchmarks/test_structured_phyre.py`):**
- Load a minimal synthetic task list (2 tasks per family, 8 total) without `data/phyre_tasks.pkl`
- All 4 conditions run without error and return accuracy as `float` in `[0, 1]`
- `l4_only` and `l4l5_full` return valid accuracy with only 2 training pairs per task (L4 no-op until ‚â•2 pairs)

**Distractor construction test:**
- Confirm all 5 candidates for a given task share an identical `test.init` snapshot (field-level equality)
- Confirm `correct_idx` is within `[0, 4]`

## Scope

- Physics simulation (pymunk) runs at generation time only. No runtime simulation during benchmarking.
- Only the 4 physics families listed above. No fluid dynamics, springs, or joints.
- No visual rendering ‚Äî all encoding operates on object state dicts, not pixel arrays.
- L4 and L5 are not modified. Any PhyRE-specific tuning (e.g. surprise thresholds) is deferred to a future sub-project.
- ARC and math benchmarks are unaffected.
